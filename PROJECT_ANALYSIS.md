# Отчет по анализу проекта "English with Mr. Sam"

## 1. Общая оценка готовности
**Статус:** ⚠️ **Критический риск / Не готов к релизу**

Несмотря на визуальную готовность и богатый функционал, проект содержит критические уязвимости, которые могут привести к полной утечке данных пользователей и захвату управления платформой в первый же день работы.

---

## 2. Критические уязвимости (Security Audit)

### 2.1. Утечка персональных данных и паролей
*   **Проблема:** Эндпоинт `/api/admin/overview` возвращает полный список пользователей, включая их **пароли в открытом виде**.
*   **Риск:** Любой скомпрометированный аккаунт администратора или ошибка доступа открывает злоумышленнику базу всех учеников.
*   **Место в коде:** `backend/server.py`, логика обработки эндпоинта `overview`.

### 2.2. Отсутствие авторизации в Admin API (Insecure Direct Object Reference)
*   **Проблема:** Многие административные эндпоинты (например, управление менторами, просмотр заявок) проверяют только параметр `username`, передаваемый в запросе, но не проверяют сессионный токен.
*   **Риск:** Любой человек, знающий `username` администратора (например, `azamat_admin`), может отправлять запросы от его имени, просто подставляя имя в URL или тело запроса.
*   **Место в коде:** `backend/server.py`, функции `is_admin_user` и соответствующие эндпоинты `do_GET`/`do_POST`.

### 2.3. Небезопасное хранение учетных данных
*   **Проблема:** В базе данных SQLite в таблице `users` пароли хранятся в открытом виде (колонка `password`) параллельно с хэшем. 
*   **Риск:** При получении доступа к файлу БД (например, через бэкап или ошибку сервера) злоумышленник получает все пароли мгновенно.

### 2.4. Хардкод секретов
*   **Проблема:** Пароли администратора и токены ботов прописаны прямо в исходном коде или хранятся в открытых `.py` файлах.
*   **Риск:** При попадании кода в публичный репозиторий или доступе к серверу данные сразу становятся известны.

---

## 3. Архитектурный анализ

*   **Backend:** Использование низкоуровневой библиотеки `http.server` для реализации сложного API. Это привело к созданию монолитного файла `server.py` (>7000 строк), который крайне сложно поддерживать и защищать.
*   **Database:** SQLite — хороший выбор для 100-300 учеников, но требует правильной настройки блокировок (WAL mode) при работе в многопоточном режиме Python.
*   **Telegram Bots:** Реализованы на чистых запросах (Long Polling). Работают стабильно, но архитектурно сильно связаны с небезопасным API сайта.

## 4. Стратегия деплоя (Production)

### Вариант 1: Google Firebase (Serverless)
*   **Кому подходит:** Если безопасность важнее всего и есть время на переписывание кода.
*   **Плюсы:** Безопасная авторизация "из коробки", отличный Hosting, бесплатный уровень.
*   **Минусы:** Нужно переписывать API под Cloud Functions, ботов нужно переводить на Webhooks.

### Вариант 2: Railway (Рекомендуемый PaaS)
*   **Кому подходит:** Идеальный баланс между скоростью запуска и стабильностью.
*   **Плюсы:** 
    *   Запуск текущего Python-кода без переписывания.
    *   Автоматический SSL (HTTPS).
    *   Поддержка Long Polling для ботов (они будут работать сразу).
    *   Легкое подключение PostgreSQL вместо SQLite.
*   **Минусы:** Платный тариф (~$5/мес), но это оправдано стабильностью.

## 5. Дорожная карта перед запуском (Checklist)

1.  **Security Fix:** Удалить колонку `password` из БД, оставить только хэши.
2.  **Auth Fix:** Внедрить обязательную проверку `session_token` во все эндпоинты `/api/admin/`.
3.  **Secrets:** Вынести все токены и пароли в переменные окружения (`.env`).
4.  **Refactoring:** Разделить `server.py` хотя бы на несколько файлов (auth, lessons, admin).
5.  **Infrastructure:** 
    *   Создать `Procfile` и `requirements.txt`.
    *   Настроить автоматический деплой через Railway.
    *   Подключить мониторинг ошибок.
